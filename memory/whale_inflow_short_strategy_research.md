## 2. 新戦略の概念

### コアロジック
```
オンチェーン大口ウォレット → 取引所へのBTC流入検知 → ショートエントリー
```

### 仮説
- 大口ウォレット（50BTC以上保有）から取引所へのBTC送金 = 売却準備
- 過去実例で「大口流入 → 数時間〜数日で価格下落」の傾向がある
- この流入イベントをシグナルにショートポジションを構築

### 既存戦略との違い
| 項目 | 既存Whaleコピー | 新: Inflow Short |
|------|----------------|-----------------|
| データソース | HL API（取引所内） | オンチェーン（blockchain） |
| 対象 | HL上のトレーダーポジション | BTC大口保有者の送金 |
| 方向 | 同方向（コピー） | 逆方向（流入=売り圧→ショート） |
| 監視対象 | 3-10ウォレット | 100ウォレット |
| API | Hyperliquid API | Blockchain API + Whale Alert等 |

## 5. リスク・課題

### 技術的課題
1. **取引所アドレスの特定**: 取引所は多数のウォレットを使い分ける
2. **誤検知**: 取引所間移動（コールド→ホット）を売り圧と誤認
3. **レース条件**: 大口流入検知→エントリーまでの遅延
4. **レート制限**: 無料APIでのリアルタイム監視は困難

### 市場リスク
1. **流入≠売却**: コールドウォレット間の移動の可能性
2. **ヘッジ目的**: 売却ではなくデリバティブ取引の担保かも
3. **価格反応の不確実性**: 50BTC流入でも価格が下がらないケース
4. **流動性リスク**: 小口（$190）でのショートはスリッページ大

### 統計的課題
1. **サンプルサイズ**: 過去の大口流入イベントは限定的
2. **生存バイアス**: 成功例だけを見るリスク
3. **過学習**: パラメータ最適化が過去データに過剰適合

## 10. OOS検証結果と最終判定 (2026-04-19)

### 作成ファイル
- `data/btc_inflow_oos_validation.py` — IS/OOS分割検証スクリプト

### OOS検証結果（OKEx 50-500 BTC → SHORT 24h）

| 指標 | IS（前半） | OOS（後半） | 判定 |
|------|-----------|------------|------|
| 期間 | 2025-12 ~ 2026-03 | 2026-03 ~ 2026-04 | — |
| イベント数 | 27 | **2** | サンプル不足 |
| 下落率（WR） | 63.0% | 50.0% | 悪化 |
| EV/トレード | **+1.09%** | **-0.44%** | **FAIL** |

### 全テスト結果
| テスト | IS EV | OOS EV | 判定 |
|--------|-------|--------|------|
| OKEx 50-500 BTC, 24h | +1.09% | -0.44% | **FAIL** |
| OKEx 50-500 BTC, 4h | +0.23% | +0.30% | PASS（サンプル2件） |
| OKEx all sizes, 24h | +0.98% | -0.44% | **FAIL** |
| All exchanges 50-500, 24h | +0.30% | -0.23% | **FAIL** |

### 最終判定: **FAIL — 実装見送り**

#### 理由
1. **OOS検証FAIL**: IS（前半）で見つけたエッジがOOS（後半）で消失
2. **サンプル不足**: OOS期間のイベントがわずか2件（統計的に無意味）
3. **データ偏り**: mempool.space APIが最新TXのみ返す仕様 → 古いTXが取得できず、イベントが2025-12以降に偏る
4. **ノイズ多すぎ**: 内部移転、集約TX、Tether担保移動等、売り圧ではない流入が多数

#### 根本原因
- 無料API（mempool.space）では**取引所の最新TX数件**しか取得できない
- 3年分の過去データを完全に取得するには有料API（Glassnode/CryptoQuant）が必要
- 現状のデータでは統計的に有意な結論を出せない
