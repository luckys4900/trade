## 📊 エージェント1: 戦略ツール調査

### 调查视角
「GitHub に存在する高期待値戦略ツールの现实的な評価」

### 方法論
1. **スター数ソート**: 公开リポジトリの注目度
2. **実績确认**: バックテスト结果・用户报告
3. **技術的実現可能性**: 実装の艰しさ・必要な资源

### 主要发现

#### 発見1: GitHub ツールの三角形
```
Freqtrade (49.7k ★):
  ├─ Sharpe 3.89 公式例
  ├─ Hyperopt 优化机构内置
  └─ 暗号资产対応（CCXT経由）

CCXT (42.2k ★):
  ├─ 108+ 取引所统一 API
  └─ 全戦略の「接続層」

VectorBT (7.3k ★):
  ├─ Numba + Rust で 1000× 高速化
  └─ 365日 < 10秒 完了
```

#### 発見2: 統計的アービトラージが最も実績丰富
- **ArbitrageLab**: Engle-Granger / Johansen 検定实现
- **MlFinLab**: López de Prado 系の α ラベリング
- **Riskfolio-Lib**: Kelly + CVaR ポートフォリオ最適化

#### 発見3: オンチェーン戦略は GitHub に実装なし
- SOPR/MVRV の本格データは Glassnode/CryptoQuant の有料 API
- 理由: データアクセスが闭ざられている

### 判断プロセス
```
問題: Sharpe > 1.0 のオンチェーン戦略找不到
├─ 侯補群: crypto-whale-watching-app (636 ★)
├─ 評価: Coinbase 板データのみ、実用过なし
├─ 結論: 有料 API 依赖必须、GitHub 実装不可
└─ 代替案: Glassnode API 統合 + カスタムロジック
```

## 🤖 エージェント2: AIモデル調査

### 调查视角
「机械学習ベースの战略は Sharpe > 1.0 を実現できるか」

### 方法論
1. **文献調査**: arXiv 論文・FinRL Official
2. **再現性确认**: walk-forward 検証的实施状況
3. **过学習リスク評価**: 単一バックテストの危险

### 主要发现

#### 發現1: Sharpe > 1.0 の公开事例は存在しない（再現可能な范围内）
```
❌ GitHub repo (zaind-24): Sharpe 1.93
   → 単一バックテスト、walk-forward未实施
   → 过学習疑念

❌ Helformer (学術): Sharpe 925% リターン
   → 学術論文のみ、独立再現なし

❌ FinRL Official Contest 2025: Sharpe 0.28
   → 公式ベンチマーク（再現可能）だが目標未達

✅ 株式 (Columbia Ensemble): Sharpe 1.10-1.12
   → ただし BTC ではなく DJIA 対象
```

#### 発見2: 过学習の3大問題
```
1. 非定常性（Regime Shift）に极度に弱い
   - 伝統的な金融は「σ = 1年で±15%」
   - BTC は「24時間で±10%」が常態

2. 単一バックテストの危险性
   - arXiv 2209.05559 論文が警鐘
   - walk-forward + out-of-sample + Deflated Sharpe 必须

3. 取引コストの過小評価
   - Hyperliquid Maker -0.02%, Taker 0.05%
   - ファンディングレート（8時間で0.01-0.05%）
```

#### 発見3: 推奨アンサンブル
```
短期 (4h):
  N-HiTS (安定) + Adaptive TFT (市場適応)

中期 (日足):
  LightGBM (特徴量ベース)

ポジション最適化:
  Kelly 基準 (f = (Win% × AvgWin - Loss% × AvgLoss) / AvgWin)
  + CVaR 制約 (Riskfolio-Lib)
  + RL (PPO) で週次再訓練
```

### 判断プロセス
```
問題: LSTM は過学習で有名。这是回避できるか？
├─ 侯補: N-HiTS (M4コンペ勝者)
├─ 評価: LSTM より安定、 walk-forward 向いている
├─ 結論: N-HiTS + Adaptive TFT のアンサンブル
└─ リスク管理: 週次再訓練 + Sharpe < 0.5 で信号弱化
```

## ⚡ エージェント3: パフォーマンス調査

### 调查视角
「バックテスト高速化技术の現実的限界値」

### 方法論
1. **実测値収集**: 各フレームワークの公稱ベンチマーク
2. **目标値对比**: 「365日 / 2時間」という目标の妥当性
3. **コスト対効果分析**: GPU 活用の是否有意義

### 主要发现

#### 発見1: 目标「2時間」は完全にオーバースペック
```
フレームワーク           365日×1分足    パラメータスイープ(1000通り)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
VectorBT (Numba+Rust)   < 10秒        < 1分
NautilusTrader          数十秒〜数分    数分
Polars + Numba          数秒〜十秒     数十秒
Backtrader              数十分〜数時間  数時間〜日
Pure Python             数時間         数日

→ 「2時間」の制約は完全に達成、1万倍のマージン
```

#### 発見2: 推奨スタック
```
データ ETL:
  CCXT (データ取得)
  → Polars (前処理, 7-30× 高速化)
  → Parquet (キャッシュ)

研究エンジン:
  VectorBT + Numba
  → 1000パラメータ同時評価

最適化:
  Optuna (TPE/CMA-ES)
  → joblib.Parallel (マルチプロセッシング)

本番エンジン:
  NautilusTrader (Hyperliquid ネイティブ)
```

#### 発見3: GPU 活用は慎重に
```
RAPIDS cuDF:
  → 5000銘柄 20× 高速化
  → T4 GPU で $5-10/月

结论:
  - CPU スタックで十分
  - GPU は「最終本番の数百万パラメータ」でのみ検討
```

### 判断プロセス
```
問題: GPU を活用すべきか？
├─ 侯補A: RAPIDS cuDF (5000銘柄用)
├─ 侯補B: CPU + VectorBT (十分高速)
├─ 評価: 開発复杂度とコスト増加
├─ 結論: CPU + VectorBT で当面は没问题
└─ ただし: NautilusTrader の採用検討
```

```

根拠:
  ✓ 技術的に可能（複数の学術証拠）
  ✓ ツール・実装例が豊富
  ✓ パフォーマンス制約なし
  ✗ 再現困難（市場非定常性が極度に高い）
  ✗ 公開実績が限定的

推奨マインドセット:
  「Sharpe 1.0 を目指す」 = 正しい向き
  「Sharpe 1.0 を保証」= 危険な言葉
```
